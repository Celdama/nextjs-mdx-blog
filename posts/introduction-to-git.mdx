# Les bases de Git

## Introduction

Pour faire simple, tu peux te représenter Git comme un bouton de sauvegarde vraiment puissant de tes fichiers et répertoires. Officiellement, Git est un système de contrôle de version.

Une sauvegarde dans un éditeur de texte (prenons comme exemple Word) enregistre tous les mots du document dans un seul fichier. Tu n’as jamais qu’un seul enregistrement du fichier, tel que mondoc.doc, à moins que tu fasses des copies en double (c’est vraiment difficile à gérer) :

mondoc-version1.doc, mondoc-version2.doc, mondoc-version-finale.doc

En revanche, une sauvegarde dans Git enregistrer les différences entre les fichiers et les dossiers ET conserve un historique de chaque sauvegarde. En tant que développeur, Git va te permettre d’examiner l’évolution de ton projet, et de consulter ou restaurer facilement les anciennes versions de ton projet. Et une fois connecté à un réseau, Git va te permettre de pousser (push) ton projet vers Github ou d’autres alternatives : Bitbucket, GitLab pour partager et collaborer avec d’autres développeurs.

Alors que Git fonctionne sur ton ordinateur local, Github est une installation de stockage à distance sur le Web pour tous tes projets. Cela signifier qu’en apprenant Git, tu vas pouvoir présenter ton portfolio sur Github ! Toutes les entreprises de développement considèrent l’utilisation de Git comme une compétence essentielle pour les développeurs.

Le fait de disposer d’un compte Github va te permettre de prouver à ton futur employeur potentiel ce dont tu es capable.

Lis les chapitres 1.1 à 1.4 pour avoir une vision globale des différences entre les systèmes de contrôle de version:

# Mise en place de Git
<br />

Je travail presque exclusivement sur Mac, donc je vais te montrer l’installation
pour MacOS. Tu peux facilement trouver l’installation si tu es sur Windows. Je te
laisse faire tes premières recherches.

## I. Installer Git
<br />

### 1.1 Installer Homebrew

Tu dois d’abord installer Homebrew, vérifie d’abord que tu remplis les exigences pour l’installation de Homebrew <a href='https://docs.brew.sh/Installation#macos-requirements' target='_blank'>ici</a>

Ensuite copie et colle ce qui suit dans ton terminal :

<SyntaxHighlighter language='bash'>
  {`
  /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
  `}
</SyntaxHighlighter>

### 1.2 Mise à jour de Git

MacOS est déjà livré avec une version de Git, mais il faut la mettre à jour vers la dernière version.

Dans ton terminal colle ça:

<SyntaxHighlighter language='bash'>
  {`
  brew install git
  `}
</SyntaxHighlighter>
Cela va installer la dernière version de Git.

### 1.3 Vérifie ta version

Ferme ton terminal, et ouvre le à nouveau et vérifie que ta version de Git est au moins la version 2.28 en utilisant cette commande :

<SyntaxHighlighter language='bash'>
  {`
  git --version
  `}
</SyntaxHighlighter>

## II. Configurer Git et Github
<br />

### 2.1 Configuration de Git

Pour que Git fonctionne correctement, tu dois lui faire savoir qui tu es, afin qu’il puisse relier un utilisateur Git Local (toi) à Github. Lorsque tu vas travailler en équipe, cela permet aux gens de voir ce que tu as fais, et qui est l’auteur de chaque ligne de code.

Les commandes ci-dessous vont configurer Git.

Assure-toi de saisir tes propres informations et entre les guillemets (mais inclus les guillemets).

<SyntaxHighlighter language='bash'>
  {`
  git config --global user.name "Ton nom"
  git config --global user.email "tonnom@example.com"
  `}
</SyntaxHighlighter>

Github à récemment changé la branche par défaut des nouveaux dépôts de master à main.

Change la branche par défaut pour Git en utilisant cette commande:

<SyntaxHighlighter language='bash'>
  {`
  git config --global init.defaultBranch main
  `}
</SyntaxHighlighter>

Pour vérifier que tout fonctionne correctement, entre ces commandes et vérifie si le résultat correspond à ton nom et à ton adresse mail

<SyntaxHighlighter language='bash'>
  {`
  git config --get user.name
  git config --get user.email
  `}
</SyntaxHighlighter>

### 2.2 Créer un compte Github

Rend toi simplement sur le site Github et créer toi un compte !

### 2.3 Créer une clé SSH

Une clé SSH est un identifiant sécurisé par cryptage. C’est comme un mot de passe très long utilisé pour identifier ton ordinateur. Github utilise les clés SSH pour te permettre de télécharger vers ton dépôt sans avoir à taper ton nom d’utilisateur et ton mot de passe à chaque fois.

Tout d’abord il faut vérifier si tu as une clé SSH déjà installée. Tape dans le terminer et vérifier avec les informations suivantes:

<SyntaxHighlighter language='bash'>
  {`
  ls ~/.ssh/id_ed25519.pub
  `}
</SyntaxHighlighter>

Si un message apparaît dans le terminal contenant le texte “No such file or directory”, alors tu n’as pas encore de clé SSH Ed22519, et tu dois en créer une. Si aucun message de ce type n’apparaît, tu peux passer à l’étape 2.4.

Pour créer une nouvelle clé SSH, exécute la commande suivante dans ton terminal. L’indicateur -C suivi de ton adresse e-mail permet à Github de savoir qui tu es.

Remarque : les crochets `<>` dans l’extrait de code, indiquent que tu dois remplacer cette partie de la commande par tes propres informations. N’inclus pas les crochets eux-mêmes dans ta commande. Par exemple, si ton adresse mail c’est `monemail@yahoo.com` tu vas taper:

<SyntaxHighlighter language='bash'>
  {`
  ssh-keygen -t ed25519 -C monemail@yahoo.com
  `}
</SyntaxHighlighter>

Quand il te demande un emplacement pour enregistrer la clé générée, appuie simplement sur Entrée.

Ensuite, il va te demander un mot de passe : tu peux, mais ce n’est pas obligatoire.

### 2.4 Link ta clé SSH à Github

Tu dois maintenant indiquer à Github quelle est ta clé SSH afin de pouvoir pousser ton code sans avoir à taper un mot de passe à chaque fois.

Connecte toi à Github, et clique sur ta photo de profil dans le coin supérieur droit. Ensuite, sur paramètre dans le menu déroulant.

Ensuite, dans la partie gauche, clique sur clé SSH et GPG.

Ensuite, clique sur le bouton vert dans le coin supérieur droit qui indique Nouvelle clé SSH. Donne à ta clé un nom suffisamment descriptif pour que tu puisses te rappeler d’où elle provient. Laisse cette fenêtre ouverte pendant que tu effectues les étapes suivantes.

Tu dois maintenant copier ta clé SSH publique. Il faut que tu entres dans ton terminal

<SyntaxHighlighter language='bash'>
  {`
  cat ~/.ssh/id_ed25519.pub
  `}
</SyntaxHighlighter>

Copie la sortie de cette commande, qui commence par ssh-ed25519 et se termine par ton adresse mail.

Maintenant, retourne sur Github dans la fenêtre de ton navigateur et colle ta clé que tu as copiée dans le champ de la clé. Puis, clique sur Ajouter une clé SSH. C’est fait !

### Utiliser le workflow Git

1. Crée un nouveau fichier dans le dossier `git test` appelé `hello_world.txt` avec la commande `touch hello_world.txt`

<img className='post-img' src='/git/5.jpg' alt='Image alt' />

2. Tape `git status` dans ton terminal. Tu vas remarquer que ton fichier `hello_world.txt` est affiché en rouge, ce qui signifie que ce fichier n'est pas indexé.

<img className='post-img' src='/git/6.jpg' alt='Image alt' />

3. Tape `git add hello_world.txt` Cette commande ajoute ton fichier hello_world.txt à la staging area de Git. La zone de transit fait partie du processus en deux étapes pour effectuer un commit dans Git. Pense à la zone de transit comme une “salle d’attente” pour tes changements jusqu’à ce que tu les envois sur Github. Maintenant, tape à nouveau `git status`. Dans la sortie, tu remarques que ton fichier est maintenant en vert, ce qui signifie que ton fichier est maintenant dans la zone de transit.

<img className='post-img' src='/git/7.jpg' alt='Image alt' />

4. Tape `git commit -m "Add hello_world.txt" et ensuite tape git status une fois de plus. La sortie devrait maintenant dire : "nothing to commit, working tree clean", indiquant que tes changements ont été commit.

<img className='post-img' src='/git/8.jpg' alt='Image alt' />

5. Le message “Your branch is ahead of ‘origin/main’ by 1 commit” signifie simplement que tu as maintenant des changements plus récents que ce qui se trouve sur ton repo Github distant.

6. Tape `git log`et regarde la sortie. Tu devrais voir une entrée pour ton commit “Add hello_world.txt”. Tu verras également des détails sur l’auteur qui a fait le commit et la date et l’heure du commit. Si ton terminal est bloqué dans un écran avec (END) en bas, appuis simplement sur “q” pour sortir.

<img className='post-img' src='/git/9.jpg' alt='Image alt' />

## Cheatsheet
<br />

Voici une liste des commandes Git les plus couramment utilisées. Essaye de te familiariser avec ces commandes afin de pouvoir éventuellement t’en souvenir :

Commandes relatives à un repo distant:

- `git clone git@github.com:USER-NAME/REPOSITORY-NAME.git`
- `git push` ou `git push origin main` (Les deux accomplissent le même objectif dans ce contexte)

Commandes relatives au workflow:

- `git add .`
- `git commit -m "Un message qui décrit ce que tu viens de faire"`

Commandes relatives à la vérification de l’état ou de l’historique

- `git status`
- `git log`

La syntax basic de Git est `program | action | destination`

Par exemple :

- `git add .` est compris en tant que `git | add | .` où la destination représente tout ce qu’il y a dans le répertoire courant.
- `git commit -m "message"` est compris en tant que `git | commit -m | "message"`
- `git status` est compris en tant que `git | status | (no destination)`

## Meilleures pratiques Git
<br />

Il y a beaucoup à apprendre sur l’utilisation de Git. Git n’est pas seulement utile lorsque tu collabores avec d’autres personnes. Il est également utile lorsque tu travailles de manière indépendante.

Deux bonnes pratiques à considérer sont les commits atomiques et l’utilisation de ces commits atomiques pour rendre tes messages de commit plus utiles aux futurs collaborateurs.

Un commit atomique est un commit qui inclut des changements liés à une seule fonctionnalité ou tâche de ton programme. Il y a deux raisons principales à cela : d’abord, si un changement que tu apportes s’avère causer des problèmes, il est facile de revenir sur ce changement spécifique sans perdre les autres changements ; et ensuite, cela va te permettre d’écrire de meilleurs messages de commit, car plus spécifique.

Les commandes vues dans cet article sont celles que tu utiliseras le plus souvent avec Git. Ne t’inquiète pas si tu ne connais pas toutes les commandes. A force de les utiliser, cela va rentrer tout seul. Pour tous tes futurs projets, prends l’habitude de créer un repo github, et de push tes modifications de manières atomique !
